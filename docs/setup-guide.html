Setup How-To

A complete, step-by-step guide from zero to live signal routing. Pick one hosting option: Part A — Render (recommended; gives you the HTTPS URL TradingView needs) or Part B — self-host on Linux. Then everyone does Parts C–H. Lines starting with $ are typed in a terminal (don't type the $). Stay on a Demo account until test signals fill correctly.

A · Deploy on Render B · Self-host (Linux) C · Configure D · TradingView E · Test safely F · Go live G · Operate & update H · Troubleshooting

Part A · Deploy on Render recommended

TradingView only sends webhooks to ports 80/443. Render serves your app over HTTPS (443) with a public URL, so it's the simplest way to receive alerts. This repo ships a render.yaml blueprint that sets everything up for you.

A1 · Before you start

  1. 1
    Create accounts (all have free tiers to begin): GitHub, Render, and a Tradovate account.
  2. 2
    Make sure the code is in a GitHub repo Render can access — your own repo or a fork of it. Render deploys from GitHub.
  3. 3
    Decide a strong dashboard password and a strong webhook secret now (you'll paste them in shortly).

A2 · Create the service (Blueprint)

  1. 1
    In Render, click New → Blueprint.
  2. 2
    Select your GitHub repository and confirm. Render reads render.yaml and proposes a web service + a 1 GB persistent disk at /var/data.
  3. 3
    Choose the Starter instance type (always-on). The free plan sleeps after ~15 min idle, so the first alert after a quiet period can be delayed or missed — not suitable for live trading.
  4. 4
    Click Apply / Create. Don't open the URL yet — finish A3 and A4 first.
Prefer to set it up by hand? New → Web Service → pick the repo → Build command pip install -r requirements.txt, Start command python run.py, Health check path /healthz. Then do A3 and A4 manually.

A3 · Set environment variables

Render → your service → Environment. Add:

KeyValueWhy
NEXUSPRED_DATA_DIR/var/dataStore settings/tokens on the disk (must match A4's mount path)
DASHBOARD_PASSWORDyour passwordProtects the public dashboard (HTTP Basic auth)
PYTHON_VERSION3.11.9Avoids a too-new Python with no prebuilt wheels (build fails otherwise)
Do not set PORT — Render injects it and the app binds it automatically.

A4 · Attach a persistent disk

⚠ Do this before you enter any Tradovate token. Without a disk, settings save to a temporary folder and are wiped on the next deploy (and you'll see a “Permission denied: /var/data” error if the env var points at a non-existent path).
  1. 1
    Render → your service → Disks → Add Disk (Blueprint users already have one — verify it).
  2. 2
    Set: Name nexuspred-data, Mount Path /var/data (exactly equal to NEXUSPRED_DATA_DIR), Size 1 GB.
  3. 3
    Save — Render redeploys automatically.

A5 · Deploy & first login

  1. 1
    Wait for the deploy to finish (Logs show Uvicorn running). You get a URL like https://nexuspred.onrender.com.
  2. 2
    Open it. A login box appears — enter any username and your DASHBOARD_PASSWORD (only the password is checked).
  3. 3
    Continue with Part C below to configure Tradovate. ✅

A6 · Render build/runtime issues

  1. ?
    Build fails compiling pydantic-core/orjson → Python too new. The repo pins 3.11 via .python-version; if your service predates that, set PYTHON_VERSION=3.11.9 and Manual Deploy → Clear build cache & deploy.
  2. ?
    “Permission denied: /var/data” / 500 on save → the disk isn't mounted at /var/data. Do A4, then redeploy.
  3. ?
    First signal after idle is missed → you're on the free plan (sleeps). Switch to Starter.
On Render, the in-app Update button is disabled — updates deploy automatically when you push to GitHub (or click Manual Deploy). Skip Part B and go to Part C.

Part B · Self-host on Linux alternative

Only needed if you're not using Render. Commands use the folder /home/py/nexuspred. TradingView still needs HTTPS/443 — see B7. On Windows? Install Python 3.9+ (tick “Add python.exe to PATH”) and Git, then run install.bat and start.bat; the rest is identical.

B1 · Prerequisites

  1. 1
    Open a terminal (or SSH in: $ ssh py@YOUR_SERVER_IP).
  2. 2
    Install Python, venv, pip and git. 
    # Debian / Ubuntu
$ sudo apt update
$ sudo apt install -y python3 python3-venv python3-pip git

# Fedora / RHEL
$ sudo dnf install -y python3 python3-pip git
  3. 3
    Verify: $ python3 --version and $ git --version.

B2 · Download the code

  1. 1
    Clone into /home/py/nexuspred (clone, not ZIP, so the Update button works): 
    $ cd /home/py
$ git clone https://github.com/tobiasgiger/nexuspred.git
$ cd /home/py/nexuspred
  2. 2
    Already have a ZIP in that folder? Convert it once: $ ./connect-git.sh

B3 · Install

  1. 1
    Run the installer (creates an isolated .venv and a start.sh): 
    $ cd /home/py/nexuspred
$ chmod +x install.sh
$ ./install.sh
  2. 2
    Wait for “Installation complete!”.

B4 · Start

$ cd /home/py/nexuspred
$ ./start.sh        # serves on 0.0.0.0:9000 — Ctrl+C to stop

B5 · Keep it running after you close SSH

A bare ./start.sh stops when you disconnect. Pick one:

  1. A
    systemd (recommended) — survives disconnect, reboot, crashes: 
    $ ./install.sh --service
$ sudo loginctl enable-linger py     # key: keeps it alive after logout
    Manage: $ systemctl --user status tradovate-bridge · restart · $ journalctl --user -u tradovate-bridge -f
  2. B
    tmux: $ tmux new -s bridge → run ./start.sh → detach with Ctrl+b then d → reattach $ tmux attach -t bridge.
  3. C
    nohup: $ nohup ./start.sh > bridge.log 2>&1 & (stop: $ pkill -f run.py).

B6 · Open / whitelist port 9000

  1. 1
    Confirm it's listening on all interfaces (expect 0.0.0.0:9000): $ ss -tlnp | grep 9000
  2. 2
    Open the OS firewall: 
    $ sudo ufw allow 9000/tcp                 # Ubuntu/Debian
$ sudo firewall-cmd --permanent --add-port=9000/tcp && sudo firewall-cmd --reload   # Fedora/RHEL
  3. 3
    Cloud server? Also add an inbound TCP 9000 rule in the provider firewall: AWS Security Group, GCP VPC Firewall, Azure NSG, Oracle Security List, Hetzner/DO/Vultr Firewall.
  4. 4
    Test: $ curl http://localhost:9000/healthz (on the server) and $ curl http://YOUR_SERVER_IP:9000/healthz (remote). Local works but remote hangs = firewall.

B7 · Make it reachable for TradingView (HTTPS)

TradingView needs a public HTTPS URL. Self-hosting on port 9000 isn't enough — put a tunnel in front:

# Cloudflare Tunnel (free, no port-forward)
$ cloudflared tunnel --url http://localhost:9000

# or ngrok
$ ngrok http 9000
Use the https://… address it prints for your webhook in Part D.

Part C · Configure the bridge

Done in the dashboard (your Render URL, or http://YOUR_SERVER_IP:9000). Open the Settings tab.

C1 · Set a dashboard password

  1. 1
    On a public host this is essential. On Render it's the DASHBOARD_PASSWORD env var (already set in A3). Self-hosting publicly? Set Settings → Webhook Security → Dashboard password.

C2 · Add your Tradovate token accounts

nexuspred is token-only — there is no username/password. Each account is authenticated by its own Tradovate access token. Go to Settings → Token Accounts and add one row per account:

  1. 1
    Click Add account. Give it a Name, pick the Environment (Demo for now), and toggle On.
  2. 2
    Paste the account's accessToken into Access token (and optionally its mdAccessToken into Check token). Tokens are auto-renewed via renewaccesstoken — no password is ever needed.
  3. 3
    Optionally set a Qty × multiplier per account (e.g. 2 doubles every order size on that account).
  4. 4
    Repeat for each account you want to mirror to, then click Save. Every signal is sent to all enabled accounts in parallel.

C3 · Connect & verify

  1. 1
    Click Connect & Verify. The top-right indicator turns green “Connected” and your accounts load.
  2. 2
    Trouble? Open Monitor → Connection Health and the Logs tab for the exact error.

C4 · Enable accounts

  1. 1
    Settings → Accounts. Tick Enabled for every account to trade — each signal goes to all enabled accounts; disabled ones are ignored.
  2. 2
    (Optional) Set a Qty × multiplier per account, then Save Accounts.

C5 · Symbol mapping

  1. 1
    Settings → Current Symbol Mapping. Confirm each TradingView symbol points to the right Tradovate contract (e.g. MNQ1! → MNQU6).
  2. 2
    After every rollover, update the contract here (e.g. MNQU6 → MNQZ6) and click Save Mapping. No restart needed.

C6 · Trading rules & safety

Defaults match the strategy (editable under Settings → Trading Rules):

SignalOrderTypeQty
buy / sellEntryMarket3 (default)
tp1 / tp2 / tp3Take-profitLimit1 each
slStop-lossStopfull position
move_slMove the stop——
close_allCancel + flattenMarketfull position
🔒 Trading enabled is OFF by default. Leave it off until you've tested (Part E).

Part D · Connect TradingView

D1 · Your webhook URL

  1. 1
    Set a strong Webhook secret in Settings and Save.
  2. 2
    Your URL (also shown on the Test & Webhook tab): 
    https://YOUR-SERVICE.onrender.com/webhook/YOUR_SECRET
# self-host:  http://YOUR_HOST:9000/webhook/YOUR_SECRET (front with HTTPS — see B7)

D2 · Create the alert

  1. 1
    In TradingView, create an alert on your strategy/symbol.
  2. 2
    Under Notifications, enable Webhook URL and paste your URL.
  3. 3
    If you set a passphrase in Settings, include "passphrase":"…" in the alert JSON.

D3 · Alert message (signal reference)

Set the alert message to the strategy JSON. The bridge routes on the action field:

actionexample eventsdoes
buy / sellentryMarket entry + TP/SL bracket
move_sltp1_hitMoves the stop to new_sl
trail_activetp2_hitAcknowledged (strategy keeps sending move_sl)
close_alltp3_hit, sl_hit, runner_exitCancels orders + flattens
 
{"event":"entry","action":"sell","symbol":"MNQ1!",
 "entry":30267,"sl":30285.07,
 "tp1":30261.58,"tp2":30265.19,"tp3":30247.04}

Part E · Test safely

E1 · Simulator (no real orders)

  1. 1
    Open the Simulator tab, pick a scenario, click Run all (or step through). Orders are filled in memory — nothing reaches Tradovate, no credentials needed.

E2 · Test & Webhook

⚠ The Test & Webhook tab runs the live pipeline and places real orders if Trading is enabled. Use it on Demo, or keep Trading off to just see routing.

Part F · Go live

F1–F3 · Switch to live

  1. 1
    Verify demo fills on Monitor (Open Positions, Recent Orders) and Logs.
  2. 2
    Settings → switch Environment to Live, re-enter live credentials, Connect & Verify, and re-check Accounts.
  3. 3
    Turn Trading enabled ON. You're live. 🚀

Part G · Operate & update

G1 · Monitor

  1. 1
    Monitor tab: connection health (status, token expiry, last renew), open positions, active trades, recent orders.
  2. 2
    Logs tab: every signal and event with timestamps.

G2 · Updates

  1. 1
    Render: push to GitHub → auto-deploy (or Manual Deploy).
  2. 2
    Self-host: click Update available in the header, or $ cd /home/py/nexuspred && git pull origin main then restart.

Part H · Troubleshooting

  1. ?
    Can't reach the dashboard (self-host) — open TCP 9000 in the OS + cloud firewall (B6); verify $ curl http://localhost:9000/healthz.
  2. ?
    Render build fails / 500 on save — see A6 (Python pin, disk at /var/data).
  3. ?
    Stops after SSH disconnect — run as a service + enable-linger (B5).
  4. ?
    Disconnected / signal didn't go through — check Connection Health; the token auto-renews and webhooks re-authenticate on demand. Keep the host always-on (Starter plan / systemd) so renewal runs.
  5. ?
    “Update failed: Not a git checkout” — you used a ZIP; run $ ./connect-git.sh once.
  6. ?
    Signal rejected “not mapped” — add the symbol in Current Symbol Mapping (C5).
  7. ?
    Nothing trades — is Trading enabled on and at least one account Enabled?
