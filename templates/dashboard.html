    ⚡ Quick install (copy-paste all at once)
    
        
    In a hurry? Paste this whole block into the terminal (as the
          py user). Each line is explained in the detailed steps below.
    
        
    $ sudo apt update
$ sudo apt install -y python3 python3-venv python3-pip git
$ cd /home/py
$ git clone https://github.com/tobiasgiger/nexuspred.git
$ cd /home/py/nexuspred
$ chmod +x install.sh
$ ./install.sh --service          # installs + auto-starts on boot
$ sudo loginctl enable-linger py  # keep running after you log out
$ sudo ufw allow 9000/tcp         # open the firewall (if ufw is active)
    
        Then open http://YOUR_SERVER_IP:9000. If a cloud
          server (AWS/GCP/Azure/Oracle/Hetzner…), you must also open
          TCP 9000 in the provider's firewall — see step 6.
      
    

      
    
        
    ☁️ Deploy on Render (recommended for TradingView)
    
        
    TradingView only sends webhooks to port 80/443.
          Render gives every service a public HTTPS (443) URL, so it's the
          easiest way to receive alerts without touching your own server's port 80. This
          repo ships a render.yaml blueprint. Steps 7–15 below still apply
          (done in the dashboard at your Render URL).
    
        
      
          
    1. 1
      Make sure your GitHub repo is reachable by Render (your own repo or a fork). Push any changes you want deployed.
      2. 
          
    2. 2
      In Render: New → Blueprint, pick this repository. Render reads render.yaml and proposes a web service + a 1 GB persistent disk at /var/data.
            Prefer manual setup? New → Web Service → Build: pip install -r requirements.txt, Start: python run.py, Health check path: /healthz, add a disk mounted at /var/data.
          
      3. 
          
    3. 3
      Set environment variables:
            
      NEXUSPRED_DATA_DIR = /var/data      # persists settings/tokens (disk)
DASHBOARD_PASSWORD =   # protects the public dashboard
      
            Optional: TRADOVATE_ACCESS_TOKEN / TRADOVATE_CHECK_TOKEN. Render injects PORT automatically — don't set it.
          
      4. 
          
    4. 4
      Choose the Starter plan (always-on). The free plan sleeps after ~15 min idle, so the first alert after a quiet spell can be delayed or missed — not ideal for trading.
      5. 
          
    5. 5
      Click Deploy. When it's live you get a URL like https://nexuspred.onrender.com.
      6. 
          
    6. 6
      Open that URL, log in (any username + your DASHBOARD_PASSWORD), then configure everything from step 8 below (authenticate, accounts, symbol map, webhook secret).
      7. 
          
    7. 7
      Your TradingView webhook URL is simply:
            
      https://YOUR-SERVICE.onrender.com/webhook/YOUR_SECRET
      
            HTTPS on 443 — no tunnel or port-forwarding needed. 🎉
          
      8. 
          
    8. 8
      Updates on Render are automatic: push to GitHub (or click Manual Deploy in Render). The in-app Update button is disabled there.
      9. 
        
    
        
    🔐 The dashboard is now on the public internet — always set DASHBOARD_PASSWORD, use a strong Webhook secret, and keep Trading enabled off until you've tested.
    
        
    🖥️ Want to self-host on your own Linux box instead? Follow the steps below. (TradingView still needs 443 — use the Cloudflare Tunnel / ngrok option in step 13.)
    
      
    

      
    
        
    1 · Prepare your Linux machine
    
        
    You need Python 3.9+, pip,
          venv and git. Install them once.
    
        
      
          
    1. 1
      Open a terminal (on the server, or via SSH: ssh py@YOUR_SERVER_IP).
      2. 
          
    2. 2
      Install the prerequisites.
            
      Debian / Ubuntu:
      
            
      $ sudo apt update
$ sudo apt install -y python3 python3-venv python3-pip git
      
            
      Fedora / RHEL:
      
            
      $ sudo dnf install -y python3 python3-pip git
      
          
      3. 
          
    3. 3
      Check the versions (both should print a number):
            
      $ python3 --version
$ git --version
      4. 
        
    
        
    🪟 On Windows instead? Install
          Python 3.9+
          (tick “Add python.exe to PATH”) and
          Git for Windows,
          then double-click install.bat and start.bat. The rest
          of this guide (from step 6) is identical.
    
      
    

      
    
        
    2 · Get the code into /home/py/nexuspred
    
        
    Use git clone (not a ZIP) so the dashboard
          Update button works later.
    
        
      
          
    1. 1
      Go to your home folder and clone the repo:
            
      $ cd /home/py
$ git clone https://github.com/tobiasgiger/nexuspred.git
      
            This creates /home/py/nexuspred with all the files.
          
      2. 
          
    2. 2
      Enter the folder (you'll run everything from here):
            
      $ cd /home/py/nexuspred
      3. 
        
    
        
    📦 Already downloaded a ZIP into /home/py/nexuspred?
          Turn it into a git checkout once so updates work:
          
    $ cd /home/py/nexuspred
$ ./connect-git.sh
    
      
    

      
    
        
    3 · Install
    
        
    The installer builds an isolated Python environment
          (.venv) and writes a start.sh launcher. Nothing
          else on your system is changed.
    
        
      
          
    1. 1
      Make the installer runnable and run it:
            
      $ cd /home/py/nexuspred
$ chmod +x install.sh
$ ./install.sh
      2. 
          
    2. 2
      Wait for “Installation complete!”. If it says Python is missing, redo step 1.
      3. 
        
    
        
    ⚙️ Want it to start automatically on boot and pick a port in
          one go? Use the flags: $ ./install.sh --service --port 9000
          (more on the service in step 5).
    
      
    

      
    
        
    4 · Start the bridge
    
        
      
          
    1. 1
      Start it from the project folder:
            
      $ cd /home/py/nexuspred
$ ./start.sh
      
            You'll see “Uvicorn running on http://0.0.0.0:9000”. Leave this terminal open — closing it stops the bridge (step 5 fixes that).
          
      2. 
          
    2. 2
      To stop it, press Ctrl + C in that terminal.
      3. 
        
    
      
    

      
    
        
    5 · Keep it running after you close SSH
    
        
    If you just run ./start.sh over SSH, it stops the moment
          you disconnect. Use one of these so it keeps running (and restarts on reboot).
          Option A is recommended.
    
        
      
          
    1. A
      systemd service — survives disconnect, reboot, and crashes.
            
      $ cd /home/py/nexuspred
$ ./install.sh --service
$ sudo loginctl enable-linger py    # <-- key: keeps it alive after logout
      
            Without enable-linger, a user service is killed when you log out — that's the usual “it stops after SSH disconnect” cause.
            Manage it any time:
            
      $ systemctl --user status tradovate-bridge
$ systemctl --user restart tradovate-bridge
$ journalctl --user -u tradovate-bridge -f      # live logs (Ctrl+C to exit)
      
          
      2. 
          
    2. B
      tmux — a terminal that stays alive after disconnect (easy to watch logs):
            
      $ sudo apt install -y tmux
$ tmux new -s bridge          # opens a session
$ cd /home/py/nexuspred && ./start.sh
# detach (leaves it running): press  Ctrl+b  then  d
# later, reattach:            $ tmux attach -t bridge
      
          
      3. 
          
    3. C
      nohup — fire-and-forget (runs until reboot):
            
      $ cd /home/py/nexuspred
$ nohup ./start.sh > bridge.log 2>&1 &
      
            Watch logs: $ tail -f bridge.log  ·  Stop it: $ pkill -f run.py
          
      4. 
        
    
      
    

      
    
        
    6 · Open / whitelist port 9000
    
        
    Can't reach the dashboard from another computer? The bridge is
          almost certainly running fine — the port is being blocked. Work through these
          in order.
    
        
      
          
    1. 1
      Confirm the bridge is listening on all interfaces (you should see 0.0.0.0:9000):
            
      $ ss -tlnp | grep 9000
      
            If it shows 127.0.0.1:9000 instead, start it with HOST=0.0.0.0 ./start.sh (the default already binds 0.0.0.0).
          
      2. 
          
    2. 2
      Open the OS firewall (run the one your system uses):
            
      Ubuntu / Debian (ufw):
      
            
      $ sudo ufw allow 9000/tcp
$ sudo ufw status
      
            
      Fedora / RHEL / CentOS (firewalld):
      
            
      $ sudo firewall-cmd --permanent --add-port=9000/tcp
$ sudo firewall-cmd --reload
      
            
      Plain iptables:
      
            
      $ sudo iptables -A INPUT -p tcp --dport 9000 -j ACCEPT
      
          
      3. 
          
    3. 3
      Open the cloud provider firewall (the #1 reason a cloud server “can't connect”). Add an inbound rule allowing TCP 9000 from your IP (or 0.0.0.0/0 to allow all):
            
        
              
      • AWS EC2 → Security Group → Inbound rules
        • 
              
      • Google Cloud → VPC Network → Firewall → Create rule
        • 
              
      • Azure → Network Security Group → Inbound security rules
        • 
              
      • Oracle Cloud → VCN → Security List → Ingress rules
        • 
              
      • Hetzner / DigitalOcean / Vultr → Firewall in the panel
        • 
            
      
          
      4. 
          
    4. 4
      Test it. From the server itself (should print JSON):
            
      $ curl http://localhost:9000/api/status
      
            From another computer (replace with your server's public IP):
            
      $ curl http://YOUR_SERVER_IP:9000/api/status
      
            If the local curl works but the remote one hangs, it's a firewall (step 2 or 3).
          
      5. 
        
    
        
    🔓 Exposing 9000 to 0.0.0.0/0 puts the dashboard
          on the open internet. Prefer allowing only your own IP, or keep it closed and
          reach TradingView via a tunnel (step 13) instead of opening the port.
    
      
    

      
    
        
    7 · Open the dashboard
    
        
      
          
    1. 1
      On the same machine: open
            http://localhost:9000.
      2. 
          
    2. 2
      From another computer: http://YOUR_SERVER_IP:9000
            (find the IP with $ hostname -I, or your public IP from the cloud panel).
            Doesn't load from another computer? That's the firewall — do step 6 above.
      3. 
        
    
        
    You're reading this guide in the dashboard — so it's already open. 🎉 The tabs are along the top: Monitor, Settings, Logs, Test & Webhook, Simulator.
    
      
    

      
    
        
    8 · Authenticate with Tradovate
    
        
    Open the Settings tab → Tradovate Connection.
          Leave Environment on Demo to start. Choose one method:
    
        
      
          
    1. 1
      Username & password (easiest) — type your Tradovate login. Keep “Use web-trader fallback” ON so it works without a paid API subscription. (Have an API key? Open Advanced and add cid/sec.)
      2. 
          
    2. 2
      Access token (optional) — paste an existing accessToken under “Use an existing access token”, or set environment variables before starting:
            
      $ export TRADOVATE_ACCESS_TOKEN="your-token"
$ export TRADOVATE_CHECK_TOKEN="your-check-token"   # optional
$ ./start.sh
      
            The token is used first and auto-renewed; it falls back to your username/password if it can't be renewed.
          
      3. 
          
    3. 3
      Click Save Settings.
      4. 
        
    
        
    ⚠ Always begin on Demo. Switch to Live only after you've watched test orders fill.
    
      
    

      
    
        
    9 · Connect & choose accounts
    
        
      
          
    1. 1
      Still in Settings, click Connect & Verify. The dot in the top-right turns green “Connected” and your accounts load.
      2. 
          
    2. 2
      Scroll to the Accounts card. Tick Enabled for every account you want to trade. Each signal is sent to all enabled accounts; disabled ones are skipped.
      3. 
          
    3. 3
      (Optional) Set a Qty × multiplier per account (e.g. 2 doubles the contracts for that account). Click Save Accounts.
      4. 
        
    
        
    🔌 Not connecting? Open the Monitor tab → Connection Health and the Logs tab to see the exact error.
    
      
    

      
    
        
    10 · Check the symbol mapping
    
        
      
          
    1. 1
      In Settings → Current Symbol Mapping, confirm each TradingView symbol points to the right Tradovate contract (e.g. MNQ1! → MNQU6).
      2. 
          
    2. 2
      After every contract rollover, edit the contract here (e.g. MNQU6 → MNQZ6) and click Save Mapping. No restart needed.
      3.

    12 · Test safely (no real orders)
    
        
      
          
    1. 1
      Open the Simulator tab. Pick a scenario (e.g. Winning trade — SELL MNQ).
      2. 
          
    2. 2
      Click Run all (or Run next step) and watch the simulated entry, TP/SL, and close. Nothing is sent to Tradovate — credentials aren't even required.
      3. 
          
    3. 3
      When you're happy, you can also try Test & Webhook — but note that one runs the live pipeline and places real orders if trading is enabled.
      4. 
        
    
      
    

      
    
        
    13 · Connect TradingView
    
        
      
          
    1. 1
      In Settings, set a strong Webhook secret and Save. Your webhook URL (from the Test & Webhook tab) looks like:
            
      http://YOUR_HOST:9000/webhook/YOUR_SECRET
      2. 
          
    2. 2
      Make the bridge reachable from the internet — TradingView can't reach localhost. Pick one:
            
      # a) Cloudflare Tunnel (free, no port-forward)
$ cloudflared tunnel --url http://localhost:9000

# b) ngrok
$ ngrok http 9000
      
            Use the public https://… address they print, plus /webhook/YOUR_SECRET. (Or port-forward 9000 on your router / use a VPS public IP.)
          
      3. 
          
    3. 3
      In TradingView: create an alert → Notifications → enable Webhook URL → paste your public URL.
      4. 
          
    4. 4
      Set the alert message to the strategy JSON, for example:
            
      {"event":"entry","action":"sell","symbol":"MNQ1!",
 "entry":30267,"sl":30285.07,
 "tp1":30261.58,"tp2":30265.19,"tp3":30247.04}
      5. 
        
    
      
    

      
    
        
    14 · Go live
    
        
      
          
    1. 1
      Confirm demo fills on Monitor (positions/orders) and Logs.
      2. 
          
    2. 2
      In Settings, switch Environment to Live, re-enter live credentials, and click Connect & Verify again. Re-check the Accounts list.
      3. 
          
    3. 3
      Turn Trading enabled ON. You're live. 🚀
      4.

    16 · Troubleshooting
    
        
      
          
    1. ?
      Can't connect from another computer — open TCP 9000 in the OS firewall and your cloud provider's firewall (step 6). Verify with $ curl http://localhost:9000/api/status on the server.
      2. 
          
    2. ?
      Stops when I close SSH — run it as a service and enable lingering (step 5A: ./install.sh --service then sudo loginctl enable-linger py), or use tmux (step 5B).
      3. 
          
    3. ?
      “Update failed: Not a git checkout” — you used a ZIP. Run $ ./connect-git.sh once (step 2).
      4. 
          
    4. ?
      Can't connect to Tradovate — check Connection Health on Monitor and the Logs tab; verify credentials and Demo/Live.
      5. 
          
    5. ?
      Signal rejected / “not mapped” — add the symbol in Current Symbol Mapping (step 10).
      6. 
          
    6. ?
      Nothing trades — is Trading enabled on, and at least one account Enabled?
      7. 
          
    7. ?
      Service logs: $ journalctl --user -u tradovate-bridge -f (or the Logs tab in the dashboard).
